# Plan 59-01: Cursor Pagination & Infinite Scroll

<objective>
After this plan, the library page loads the first 50 clips immediately and scrolling to the bottom loads the next 50 without a full page reload or URL change. The backend uses cursor-based pagination keyed on created_at for stable, performant paging.

Purpose: Eliminate loading ALL clips at once, which gets slow as the library grows.
Output: Cursor-paginated /all-clips endpoint + frontend infinite scroll.
</objective>

<task id="1">
**Backend: Add cursor-based pagination to list_all_clips**

In `app/api/library_routes.py`, modify the `list_all_clips` endpoint:

1. Add a new optional query parameter: `cursor: Optional[str] = Query(default=None)` — this is an ISO 8601 timestamp string
2. When `cursor` is provided, add `.lt("created_at", cursor)` to the Supabase query BEFORE `.order("created_at", desc=True).limit(limit)`
3. When `cursor` is NOT provided, behave exactly as today (first page)
4. In the response, compute `next_cursor`: if the result has exactly `limit` items, set `next_cursor` to the `created_at` value of the LAST item in the response. If fewer items returned, set `next_cursor` to `null` (no more pages)
5. Add `next_cursor` and `has_more` (boolean) to the response dict alongside existing `clips`, `total`, `limit`, `offset`
6. Keep the `offset` parameter working for backward compatibility — if `cursor` is provided, `offset` is ignored
7. The `count="exact"` call should NOT include the cursor filter — it must return the total count of all clips for the profile

**Concrete change location:** The `list_all_clips` function starting at line ~1515 of `app/api/library_routes.py`.

**Verification:** `GET /api/v1/library/all-clips?limit=2` returns 2 clips + a `next_cursor` + `has_more: true` (assuming >2 clips exist).
</task>

<task id="2">
**Frontend: Implement infinite scroll on library page**

In `frontend/src/app/librarie/page.tsx`:

1. Add state variables: `nextCursor` (string | null), `hasMore` (boolean, default true), `loadingMore` (boolean, default false)
2. Modify `fetchAllClips` to accept an optional `cursor` parameter:
   - When cursor is null/undefined: reset clips to the response (first page load)
   - When cursor is provided: APPEND response clips to existing `clips` state using `setClips(prev => [...prev, ...data.clips])`
   - Update `nextCursor` from `data.next_cursor`
   - Update `hasMore` from `data.has_more`
3. The initial `useEffect` call remains the same (calls `fetchAllClips()` without cursor)
4. Add a `fetchNextPage` function that calls `fetchAllClips(nextCursor)` — guard with `if (!hasMore || loadingMore) return`
5. Add an IntersectionObserver:
   - Create a `sentinelRef = useRef<HTMLDivElement>(null)`
   - Place a `<div ref={sentinelRef} />` AFTER the clip grid (below all rendered clips)
   - In a `useEffect`, create an IntersectionObserver that calls `fetchNextPage()` when the sentinel enters the viewport
   - Disconnect observer on cleanup
6. Show a loading spinner below the grid when `loadingMore` is true
7. Show a "No more clips" or similar text when `hasMore` is false and clips.length > 0

**Verification:** Load page, see first 50 clips. Scroll down, see loading indicator briefly, then more clips appear. The URL does not change.
</task>
