  <action>
1. **layout.tsx** — Import AuthProvider from `@/components/auth-provider` and wrap the app content inside it. Place it inside ProfileProvider (AuthProvider should be the innermost provider since ProfileProvider may not depend on auth). The component tree should be: ProfileProvider > AuthProvider > NavBarWrapper + children. Since AuthProvider is a "use client" component, it can be nested inside the server layout via the existing pattern.

2. **api.ts** — Modify `apiFetch` to automatically inject the Supabase access token into every request. Since `apiFetch` is a standalone function (not a React hook), it cannot use `useAuth()`. Instead, use the Supabase browser client directly:
   - Import `createClient` from `@/lib/supabase/client`
   - At the top of `apiFetch`, if `skipAuth` is NOT true, call `createClient().auth.getSession()` to get the current session
   - If `session?.access_token` exists, add `Authorization: Bearer ${session.access_token}` to the headers object
   - If `skipAuth` is true, skip token injection entirely
   - The `createClient` call is safe because `@supabase/ssr`'s `createBrowserClient` returns a singleton (same instance each call) — no performance concern
   - Handle the case where getSession returns null (user not logged in) — just proceed without the header, the backend will return 401 if the route requires auth

3. **auth-provider.tsx** — No functional changes needed, but add a re-export convenience: export the `AuthContextType` interface so other components can type-check against it if needed. The existing signOut, refreshSession, user, session, loading are already correct.

IMPORTANT: Do NOT make apiFetch async-dependent on React context. The Supabase browser client stores tokens in cookies/localStorage and `getSession()` reads from that storage — it works outside React components.
