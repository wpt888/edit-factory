  <action>
Create `frontend/src/middleware.ts` (this is the Next.js convention — must be at `src/middleware.ts` when using the `src/` directory).

1. Define public routes that do NOT require authentication:
   ```typescript
   const PUBLIC_ROUTES = ["/login", "/signup", "/auth/callback", "/setup"];
   ```

2. Export a `middleware` function that:
   a. First calls `updateSession(request)` from `@/lib/supabase/middleware` to refresh cookies and get the response
   b. Checks if the current path is a public route (use `pathname.startsWith()` for each)
   c. Also skip middleware for Next.js internals and static assets: `/_next/`, `/favicon.ico`, files with extensions (`.png`, `.jpg`, `.svg`, `.css`, `.js`)
   d. If the route is public or a static asset, return the response from `updateSession` as-is
   e. If the route is protected, create a Supabase server client using `createServerClient` from `@supabase/ssr` with the request cookies (same pattern as the existing `updateSession` helper) and call `supabase.auth.getUser()`
   f. If `getUser()` returns an error or no user, redirect to `/login` with the current path as a `next` query param: `/login?next=${encodeURIComponent(pathname)}`
   g. If user exists, return the response from `updateSession`

3. Export a `config` with a `matcher` that excludes static files and Next.js internals:
   ```typescript
   export const config = {
     matcher: [
       // Match all routes except Next.js internals and static files
       "/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)",
     ],
   };
   ```

4. IMPORTANT: Use `createServerClient` from `@supabase/ssr` directly in the middleware (not the server.ts helper which uses `cookies()` from `next/headers` — that API is not available in middleware context). The `updateSession` helper already creates a server client with the correct cookie pattern for middleware. Modify the approach: enhance `updateSession` in `lib/supabase/middleware.ts` OR duplicate the Supabase client creation in middleware.ts. The cleanest approach is:
   - Call `updateSession(request)` first to refresh session cookies
   - Then create a second server client in middleware.ts to call `getUser()` for the auth check
   - OR better: modify `updateSession` to return both the response AND the user, avoiding a double client creation

   Best approach: Create the middleware to use `createServerClient` directly with request cookies, call `getUser()`, then handle redirect or pass-through. Also call `updateSession` for cookie refresh. Since `updateSession` already calls `getUser()` internally but doesn't expose the result, the simplest path is to inline the logic:

   ```typescript
   import { createServerClient } from '@supabase/ssr'
   import { NextResponse, type NextRequest } from 'next/server'

   const PUBLIC_ROUTES = ["/login", "/signup", "/auth/callback", "/setup"]

   export async function middleware(request: NextRequest) {
     const { pathname } = request.nextUrl

     // Skip public routes
     if (PUBLIC_ROUTES.some(route => pathname.startsWith(route))) {
       return NextResponse.next()
     }

     // Create Supabase client with cookie access
     let response = NextResponse.next({ request })
     const supabase = createServerClient(
       process.env.NEXT_PUBLIC_SUPABASE_URL!,
       process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
       {
         cookies: {
           getAll() { return request.cookies.getAll() },
           setAll(cookiesToSet) {
             cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value))
             response = NextResponse.next({ request })
             cookiesToSet.forEach(({ name, value, options }) =>
               response.cookies.set(name, value, options)
             )
           },
         },
       }
     )

     const { data: { user } } = await supabase.auth.getUser()

     if (!user) {
       const loginUrl = request.nextUrl.clone()
       loginUrl.pathname = '/login'
       loginUrl.searchParams.set('next', pathname)
       return NextResponse.redirect(loginUrl)
     }

     return response
   }
   ```

   This is the standard Supabase + Next.js middleware pattern from their docs.
