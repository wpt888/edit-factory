Create `app/repositories/sqlite_repo.py` implementing the full `DataRepository` interface using Python's `sqlite3` module. This is a large file (~800-1000 lines) following the same structure as `supabase_repo.py`.

**Database connection and initialization:**
- Use `sqlite3.connect()` with `check_same_thread=False` (FastAPI runs on multiple threads)
- Store the DB path as `{settings.base_dir}/data.db` (so `~/.editfactory/data.db` in desktop, `./data.db` in dev)
- On first instantiation, create the DB file and run `supabase/sqlite_schema.sql` to initialize all tables
- Enable WAL mode and foreign keys via PRAGMA
- Use `sqlite3.Row` row factory for dict-like access
- All methods must be thread-safe: use a threading.Lock around write operations

**Table name mapping:**
Create a class-level `_TABLE_MAP` dict that maps the "logical" table names (as used in DataRepository/SupabaseRepository) to the actual SQLite table names:
```python
_TABLE_MAP = {
    "editai_profiles": "profiles",
    "editai_feeds": "product_feeds",
    "editai_products": "products",
    "editai_publications": "editai_postiz_publications",
    "editai_elevenlabs_accounts": "elevenlabs_accounts",
    "editai_segment_products": "segment_product_associations",
    "editai_generated_images": "generated_images",
    "editai_prompt_templates": "image_prompt_templates",
    "editai_jobs": "jobs",
}
```
Create a helper `_t(name)` that returns `_TABLE_MAP.get(name, name)`.

**Helper methods (mirror SupabaseRepository pattern):**
- `_apply_filters(query_parts, params, filters)`: Build WHERE clauses from QueryFilters. Handle eq, neq, gt, lt, gte, lte, in_, is_ (NULL checks), like (use LIKE with case-insensitive), contains (JSON), not_is, or_. For `or_` filter strings (PostgREST format like `"profile_id.eq.X,profile_id.is.null"`), parse them into SQL OR conditions.
- `_build_select(table, filters)` -> (sql, params): Build a SELECT query with filters, ORDER BY, LIMIT, OFFSET.
- `_get_one(table, id_col, id_val)`: Fetch single row, return dict or None.
- `_insert(table, data)`: INSERT a row, generate UUID if "id" not in data, generate ISO8601 timestamps if "created_at"/"updated_at" not in data. Return the inserted row as dict.
- `_update(table, id_col, id_val, data)`: UPDATE a row, auto-set updated_at. Return updated row.
- `_delete(table, id_col, id_val)`: DELETE a row by PK.
- `_execute_select(table, filters)` -> QueryResult: Common pattern for list methods.

**Row conversion:**
SQLite rows need conversion: sqlite3.Row -> dict. Also, for JSON columns (stored as TEXT in SQLite), parse them back to Python objects when reading. Key JSON columns: `tts_settings`, `cloned_voices`, `video_template_settings`, `subtitle_settings`, `postiz_integration_ids`, `scripts`, `previews`, `render_jobs`, `source_video_ids`, `data` (jobs), `metadata` (api_costs), `selected_image_urls`, `pip_config`, `slide_config`, `tts_timestamps`, `tags`, `product_ids`, `integration_ids`, `collection_ids`, `summary`. Create a `_row_to_dict(row)` helper that converts sqlite3.Row to dict and parses JSON text columns.

**For JSON columns on write:** Serialize Python dicts/lists to JSON strings before INSERT/UPDATE. Create a `_serialize_json_fields(data)` helper.

**Domain implementations (all 25 domains):**
Follow the exact same method signatures as `supabase_repo.py`. Key differences:
1. `list_projects`: Default join with clip_content is NOT needed in SQLite (no PostgREST select syntax). Just return project rows. If `filters.select` contains a join reference like `"*, editai_clip_content(*)"`, ignore the join part and select `*`.
2. `list_clips`, `get_clip`, `list_clips_by_profile`: Same join handling -- for now return clip rows without nested clip_content. The routes that need clip_content will fetch it separately via `get_clip_content(clip_id)`.
3. `list_project_segments`: Default join `"*, editai_segments(*)"` -- implement as a LEFT JOIN in SQL, nesting the segment data under an `editai_segments` key in the result dict.
4. `list_associations`: Default join `"*, editai_products(*)"` -- implement as LEFT JOIN, nesting product data under `editai_products` key.
5. `list_export_presets`: Special OR logic for profile_id (include both profile-specific and global null profile_id presets).
6. `cleanup_old_jobs`: DELETE with lt filter on created_at, return count.
7. `list_jobs_by_project`: JSONB arrow operator `data->>project_id` -- in SQLite, use `json_extract(data, '$.project_id')`.
8. `upsert_progress`: Use `INSERT OR REPLACE` (project_id is PK).
9. `create_schedule_items`: Bulk insert using executemany.
10. `list_catalog_products`: Implement as a JOIN query across products + product_feeds tables (since there's no view in SQLite). Return products with feed info.
11. `list_catalog_products_grouped`: Similar JOIN with GROUP BY on a grouping column.
12. `table_query`: Generic escape hatch supporting select/insert/update/delete/upsert operations with QueryFilters. For upsert, use `INSERT OR REPLACE`. Apply the _TABLE_MAP translation.
13. `get_client()`: Return None (no raw client for SQLite -- escape hatch is table_query).

**UUID generation:** Use `str(uuid.uuid4())` for generating IDs on insert when no "id" is provided.

**Thread safety:** Use a `threading.Lock` for all write operations (insert, update, delete). Read operations can run concurrently with WAL mode.
