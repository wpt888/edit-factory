  <action>
Add a new endpoint to `app/api/catalog_routes.py`:

```python
@router.get("/products/{product_id}/images")
async def get_product_images(
    product_id: str,
    profile: ProfileContext = Depends(get_profile_context),

```

This endpoint:
1. Fetches the product from `v_catalog_products_grouped` by `id` to get its `group_key` (which may be None) and `image_link`.
2. If `group_key` exists, queries `v_catalog_products` (the non-grouped view) to get all variant rows with the same group_key pattern. Since the grouped view uses `COALESCE(group_key, gomag_product_id)`, query the base `v_catalog_products` view filtering by the product's external_id as the group key. Actually, since `v_catalog_products` doesn't expose `group_key`, use a raw SQL approach: query `uf.products_catalog` for all rows with matching `group_key`, collect distinct `image_url` values.
3. Actually, simpler approach: use `mcp__supabase__execute_sql` pattern in the code — use the supabase client to call an RPC or just query `v_catalog_products` filtering by the external_id. But the cleanest approach is:
   - First get the product from `v_catalog_products_grouped` to get `external_id` (which is `gomag_product_id`).
   - Then query `v_catalog_products` (non-grouped view) and filter where `external_id` matches OR create a simple SQL function.

**Simplest implementation**: Since `v_catalog_products` is a flat view (one row per catalog row), and we need to group by `group_key` which isn't in the view, use the Supabase `.rpc()` approach with a database function. But to keep it simple and avoid another migration:

Just return the product's own `image_link` as a single-element array for now. The `selected_image_urls` in the association table is what matters — users will paste/pick URLs. The endpoint returns `{"images": ["https://..."]}` containing the product's main `image_link`. Future enhancement can add variant images.

Actually, per ASSOC-04: "User can select which image(s) from the product gallery to use on a segment." For this to work we need to return available images. Let's create a small DB function in the migration (Task 1) and call it here.

**Revised approach** — add to the migration in Task 1 a function:

```sql
CREATE OR REPLACE FUNCTION public.get_catalog_product_images(p_product_id UUID)
RETURNS TABLE(image_url TEXT) AS $$
  SELECT DISTINCT pc.image_url
  FROM uf.products_catalog pc
  WHERE COALESCE(pc.group_key, pc.gomag_product_id) = (
    SELECT COALESCE(pc2.group_key, pc2.gomag_product_id)
    FROM uf.products_catalog pc2
    WHERE pc2.id = p_product_id
  )
  AND pc.image_url IS NOT NULL
  AND pc.image_url != ''
$$ LANGUAGE sql STABLE;
```

Then in the endpoint, call:
```python
result = supabase.rpc("get_catalog_product_images", {"p_product_id": product_id}).execute()
images = [row["image_url"] for row in (result.data or [])]
```

Return `{"product_id": product_id, "images": images}`.

If RPC fails (function doesn't exist), fall back to fetching the product's `image_link` from the grouped view and returning it as a single-element list.
