# Plan 58-02: File Storage Abstraction Layer

<objective>
After this plan, setting `FILE_STORAGE_BACKEND=supabase` routes file reads and writes through Supabase Storage without changing any render pipeline code. The default backend remains local filesystem. All final output file I/O goes through a `FileStorage` abstraction.

Purpose: Decouple file storage from local filesystem so deployment to cloud environments is possible.
Output: FileStorage abstraction with local and Supabase backends.
</objective>

**Approach:** Create a `FileStorage` abstract interface with two implementations:
1. `LocalFileStorage` ã wraps current `Path`-based file I/O (default)
2. `SupabaseFileStorage` ã uploads to Supabase Storage bucket, downloads on read

The abstraction only covers **output files** (rendered videos, thumbnails, audio). Input files (uploaded by user) and temp files (intermediate FFmpeg artifacts) stay local because FFmpeg needs local paths.

**Key insight:** FFmpeg always needs local files. The abstraction kicks in at the **publish step** ã after FFmpeg produces the output, `FileStorage.store()` uploads it.

**Note:** `pipeline_routes.py` and `assembly_routes.py` are NOT modified in this plan. Their file storage integration is deferred to 58-03 or later ã they already have their own DB-backed state.
</context>

  <action>
Create `app/services/file_storage.py` with the storage abstraction:

1. **`FileStorage` ABC** with methods: `store(local_path, remote_key) -> str`, `retrieve(remote_key, local_path) -> Path`, `get_url(remote_key) -> str`, `delete(remote_key) -> bool`, `exists(remote_key) -> bool`.

2. **`LocalFileStorage`** ã no-op implementation. `store()` returns `str(local_path)`. `retrieve()` returns `Path(remote_key)`. All operations work on local paths directly.

3. **`SupabaseFileStorage`** ã uploads to Supabase Storage bucket `editai-output`.
   - Constructor: lazy-init Supabase client via `app.db.get_supabase()`, create bucket if missing, fall back to `LocalFileStorage` if unavailable.
   - **CRITICAL: `store()` must guard against OOM.** Check file size before reading. If file > 500MB, log a warning and fall back to local storage. For files under 500MB, read and upload. This prevents reading entire large video files into memory.

   ```python
   def store(self, local_path: Path, remote_key: str) -> str:
       if not self._supabase:
           return self._fallback.store(local_path, remote_key)
       try:
           file_size = local_path.stat().st_size
           max_size = 500 * 1024 * 1024  # 500MB
           if file_size > max_size:
               logger.warning(f"File too large for Supabase upload ({file_size} bytes), using local: {local_path}")
               return self._fallback.store(local_path, remote_key)
           with open(local_path, "rb") as f:
               self._supabase.storage.from_(self.BUCKET_NAME).upload(remote_key, f.read())
           logger.info(f"Uploaded {local_path} to Supabase Storage: {remote_key}")
           return remote_key
       except Exception as e:
           logger.warning(f"Supabase Storage upload failed, using local fallback: {e}")
           return self._fallback.store(local_path, remote_key)
   ```

4. **`get_file_storage()` singleton factory** ã returns backend based on `settings.file_storage_backend`.

5. **In `app/config.py`**, add to the Settings class (after `desktop_mode` field):
   ```python
   file_storage_backend: str = "local"  # "local" or "supabase"
   ```
