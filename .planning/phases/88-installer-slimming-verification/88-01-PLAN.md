## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-88-01 | Tampering | `electron/package.json` extraResources filter | mitigate | Task 1 freezes the 8 ML exclusion patterns + adds grep assertions to the plan. Task 2's defense-in-depth step re-checks installer contents via `7z l` and fails the build if any forbidden directory appears — so even if a PR removes a pattern from the filter, the CI catches the resulting payload at runtime. |
| T-88-02 | Denial of Service (against ML-01 contract) | `.github/workflows/installer-size.yml` size threshold | mitigate | The workflow runs on every PR + push to main with `exit 1` on breach. Threshold is locked to a single source-of-truth literal (576716800 bytes). A PR that re-bundles PyTorch (~700 MB) would push the installer to ~1.2 GB → workflow fails → PR cannot merge if branch protection is configured. |
| T-88-03 | Elevation of Privilege | Malicious PR disabling the CI gate | mitigate (partial) — manual follow-up required | A PR could edit `installer-size.yml` to add `continue-on-error: true`, change the threshold, or delete the workflow. **Primary mitigation**: branch protection rule on `main` requiring the `Windows NSIS installer ≤ 550 MB` status check to pass. This is a GitHub Web UI action (Settings → Branches → Branch protection rules → main) that CANNOT be automated. Documented in Task 2's commit message as a manual follow-up, and will be listed in 88-01-SUMMARY.md alongside Phase 85's identical follow-up (still outstanding per STATE.md line 30). **Secondary mitigation**: the workflow review itself surfaces in `git diff` on any PR that touches it — reviewers should be alert to changes. |
| T-88-04 | Spoofing / Supply chain | BtbN FFmpeg download URL | accept | The workflow downloads `ffmpeg-master-latest-win64-gpl.zip` from a GitHub-hosted release (https://github.com/BtbN/FFmpeg-Builds). If BtbN's account is compromised, a malicious FFmpeg could be packed into the installer. Accepted because: (a) the same trust relationship exists for all dev workstations following the existing fetch instructions in ARCHITECTURE.md line 67; (b) Phase 84 already established this URL as the canonical Windows source; (c) defense would require SHA256-pinning, which is fragile against legitimate FFmpeg updates and out of scope for Phase 88 (could be tracked as a v14 hardening item alongside SIGN-01 / SIGN-03). |
