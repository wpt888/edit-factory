<objective>
Migrate the core project CRUD and clip generation routes from raw `get_client()` Supabase-style chained queries to repository method calls, enabling these routes to work with SQLite backend (offline mode).

Purpose: Currently all routes in library_routes.py do `supabase = repo.get_client()` which returns None for SQLite, causing every route to fail with 503. Migrating the essential project lifecycle routes to use repository methods (create_project, list_projects, etc.) makes offline CRUD possible.

Output: Core library routes work with both Supabase and SQLite backends.
</objective>

  <action>
Migrate the following routes from `get_client()` raw Supabase queries to DataRepository method calls. The repository pattern already has all needed methods — this is a mechanical translation.

**Routes to migrate (critical path for offline project lifecycle):**

1. **create_project** (line ~470):
   - Before: `supabase.table("editai_projects").insert({...}).execute()`
   - After: `result = repo.create_project({"name": ..., "profile_id": ...})`
   - Return ProjectResponse from result dict

2. **list_projects** (line ~511):
   - Before: `supabase.table("editai_projects").select("*", count="exact").eq(...).order(...).limit(...).offset(...).execute()`
   - After: `result = repo.list_projects(profile.profile_id, QueryFilters(eq={"status": status} if status else None, order_by="created_at", order_desc=True, limit=limit, offset=offset, count=True))`
   - Return projects from result.data, total from result.count

3. **get_project** (line ~537):
   - Before: `supabase.table("editai_projects").select("*").eq("id", project_id).eq("profile_id", profile_id).limit(1).execute()`
   - After: `proj = repo.get_project(project_id)` — check profile_id match manually, return 404 if not found or wrong profile

4. **update_project** (line ~571):
   - Before: `supabase.table("editai_projects").update({...}).eq("id", project_id).eq("profile_id", profile_id).execute()`
   - After: `proj = repo.update_project(project_id, update_data)`

5. **delete_project** (line ~675):
   - Before: raw client queries for clips + clip_content + project deletion
   - After:
     ```python
     clips_result = repo.list_clips(project_id)
     for clip in clips_result.data:
         _delete_clip_files(clip)
     if clips_result.data:
         clip_ids = [c["id"] for c in clips_result.data]
         repo.delete_clip_content_by_clip_ids(clip_ids)
         repo.delete_clips_by_ids(clip_ids)
     repo.delete_project(project_id)
     media_manager.delete_project_media(project_id)  # from plan 01
     ```

6. **list_project_clips** (the endpoint that lists clips for a project):
   - Use `repo.list_clips(project_id, filters)` instead of raw query

7. **verify_project_ownership** helper function (line ~301):
   - Before: `supabase.table("editai_projects").select("*").eq("id", project_id).eq("profile_id", profile_id).execute()`
   - After: Use `repo.get_project(project_id)` and check profile_id field

**Migration pattern for each route:**
```python
# BEFORE:
repo = get_repository()
supabase = repo.get_client() if repo else None
if not supabase:
    raise HTTPException(status_code=503, detail="Database not available")
result = supabase.table("editai_projects").select("*")...

# AFTER:
repo = get_repository()
result = repo.create_project({...})  # or list_projects, get_project, etc.
```

Remove the `get_client()` call and the `if not supabase` guard for each migrated route. The repository itself handles the backend — it never returns None.

**IMPORTANT constraints:**
- Only migrate the routes listed above (core project CRUD + clip listing + delete). Do NOT migrate all 30+ get_client() usages — that would be too large for this plan. The remaining routes (render flow, export, batch, etc.) can continue using get_client() for now and will work with Supabase backend. They will be migrated in future phases as needed.
- Keep backward compatibility — if a repository method raises an exception, catch it and return appropriate HTTP error.
- The `verify_project_ownership` helper is used by multiple routes. Migrating it benefits all callers.
