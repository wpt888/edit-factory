# Audit Desktop (Electron) vs. Web localhost — Edit Factory

**Data:** 2026-06-26  ·  **Branch:** `feat/mvp-remediation-w0-w2`  ·  **Metodă:** audit multi-agent (41 agenți, 6 dimensiuni, fiecare problemă verificată adversarial în cod)

## Concluzia de fond

Desktop-ul **NU este o aplicație separată** — este *exact același cod* ca web-ul localhost, cu trei comutatoare:

1. **Backend** rulează cu `DESKTOP_MODE=true` → `app/main.py:441` **nu montează** routerele web-only: `postiz`, `buffer`, `schedule`, `feed`, `image_generate`.
2. **Frontend** e build-uit cu `NEXT_PUBLIC_DESKTOP_MODE=true` → `navbar.tsx` **ascunde** linkuri (`WEB_ONLY_HREFS`).
3. **Shell-ul Electron** (`main.js`) pornește backend-ul + serverul Next standalone pe portul 3947.

**Toate cele 33 de probleme provin din NE-COORDONAREA acestor trei comutatoare.** Backend-ul taie un router, dar UI-ul care îl cheamă rămâne nemodificat → butoane moarte și 404-uri. Sau invers: navbar-ul ascunde o pagină al cărei backend e perfect funcțional → feature livrat dar inaccesibil. Plus două probleme critice de **izolare a datelor** care apar fiindcă desktop-ul folosește ACELAȘI Supabase cloud ca web-ul, dar cu auth bypass-at și fișiere stocate local.

**Verdict cantitativ:** 33 probleme confirmate · 2 false pozitive filtrate.

## Detaliu per problemă

### 1. 🔴 [CRITICAL] Desktop leagă cererile de profilul default GLOBAL (fără filtru user_id) → risc de a opera pe contul altui user din cloud

- **Dimensiune / categorie:** auth-data-vault · `data-risk` · încredere: high
- **Pe web (mergea):** get_profile_context (app/api/auth.py:339) selectează profilul default cu filtru dublu eq={"user_id": current_user.id, "is_default": True}; cu X-Profile-Id validează existența ȘI proprietatea (auth.py:362-369). Profilul e mereu al user-ului din JWT.
- **Pe desktop (acum):** În desktop_mode user-ul e hardcodat 'aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee' (auth.py:124). Când lipsește X-Profile-Id, get_profile_context interoghează profiles cu eq={"is_default": True} limit 1 — FĂRĂ user_id (auth.py:310). Cum tabelele rulează RLS-off + anon key pe cloud-ul partajat cu web-ul, query-ul întoarce PRIMUL profil is_default global, care aparține probabil unui user web real → toate operațiile pipeline/library/segments rulează pe profilul acelui user.
- **Impact:** Inconsistență gravă: /profiles/ (switcher) e scopat pe user-ul hardcodat → întoarce listă goală (niciun profil cloud nu are acel UUID), deci frontendul NU trimite X-Profile-Id (api.ts:34 citește localStorage gol), iar backendul leagă tăcut fiecare request de un profil străin. Desktop poate citi/scrie proiecte, segmente și chei API ale altui cont, în timp ce UI-ul de profil arată gol. Pierdere de izolare de date între conturi.
- **Fix sugerat:** În desktop_mode rezolvă profilul default cu același filtru user_id ca pe web (sau cu un user_id desktop dedicat seedat în cloud) și fă list_profiles / get_profile_context consistente; refuză fallback-ul la is_default global neproprietat.
- **Dovezi:** app/api/auth.py:124 (user hardcodat "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"); app/api/auth.py:295-323 (ramura desktop_mode), în special auth.py:310 (eq={"is_default": True} FĂRĂ user_id); auth.py:266-277 (ramura dev — dovada asimetriei: 273/275 filtrează pe user_id înainte de fallback-ul 277); auth.py:339 (web, cu user_id) + auth.py:362-369 (validare existență+proprietate la X-Profile-Id explicit); app/api/profile_routes.py:85-99 + app/repositories/supabase_repo.py:619-623 (list_profiles strict pe user_id); frontend/src/contexts/profile-context.tsx:71-118 (apel /profiles/ și setare condiționată PROFILE_ID); frontend/src/lib/api.ts:34-36,63 (X-Profile-Id trimis doar dacă localStorage non-gol)

### 2. 🔴 [CRITICAL] DB Supabase partajat + media stocată local per-mașină: sursele video adăugate pe web nu există pe desktop → stream/preview/render dau 'Video file not found'

- **Dimensiune / categorie:** native-paths-media · `data-risk` · încredere: high
- **Pe web (mergea):** Pe web, source_videos.file_path și segmentele indică fișiere absolute care EXISTĂ pe mașina web (unde au fost urcate/adăugate). Stream, preview-stream, extracția pentru render și player-ul de segment funcționează.
- **Pe desktop (acum):** Desktop pornește backend-ul cu DATA_BACKEND=supabase (electron/src/main.js:225) dar NU setează FILE_STORAGE_BACKEND, iar credentials.env seedat conține doar SUPABASE_URL/SUPABASE_KEY → file_storage_backend rămâne 'local' (config.py:158) și base_dir=%APPDATA%\EditFactory (config.py:28-58, :81). Rândurile din DB-ul cloud comun trimit la căi de pe mașina web, inexistente sub %APPDATA% pe desktop. Endpoint-urile de stream/preview dau 404 'Video file not found', iar render-ul sare peste segmentele cu sursă lipsă.
- **Impact:** Orice proiect/sursă video creat pe instanța web devine nevizionabil și nerenderabil pe desktop (și invers: sursele adăugate pe desktop prin calea locală fără copiere se rup pe web/altă mașină). Render-urile pot ieși cu segmente goale fără eroare vizibilă clară. Este principala ruptură funcțională desktop a dimensiunii.
- **Fix sugerat:** Fie setează FILE_STORAGE_BACKEND=supabase + MINIO_PUBLIC_URL în credentials.env/main.js și încarcă media în storage-ul cloud (paritate reală cu DB), fie tratează căile lipsă cu un mesaj/UX dedicat (re-localizare fișier) și exclude din render proiectele cu surse indisponibile, în loc de 404/skip silențios.
- **Dovezi:** electron/src/main.js:217-235 (DATA_BACKEND:'supabase', DESKTOP_MODE:'true', HOST:'127.0.0.1', fără FILE_STORAGE_BACKEND); electron/src/main.js:178-203 (seedDesktopEnv copiază doar credentials.env); electron/credentials.env:1-4 (doar SUPABASE_URL + SUPABASE_KEY); app/config.py:158 (file_storage_backend='local' default, neoverride-uit); app/config.py:28-58 + :71 + :81 (base_dir=%APPDATA%\EditFactory în desktop); app/utils.py:9-20 (normalize_path doar /mnt→drive, fără remap cross-machine); app/api/segments_routes.py:899-902 + :928 + :953 (upload salvează str(video_path) absolut sub base_dir/source_videos în DB cloud) și :785 (add-local stochează str(local_path) absolut); app/api/segments_routes.py:1085-1087 (stream → 404 'Video file not found') și :1108-1110 (preview-stream → 404); app/services/file_storage.py:93-95 (LocalFileStorage.store întoarce calea locală neschimbată; abstracția acoperă doar output-ul de render, nu sursele); app/services/assembly_service.py:1528-1531 ('Source video missing' → results[i]=None, segment sărit)
