  <name>Task 1: Fix Step 4 empty state flash (BUG-01)</name>
  <files>frontend/src/app/pipeline/page.tsx</files>
  <action>
The bug: When `handleRender` succeeds, it calls `setVariantStatuses(data.variants || [])` but the render endpoint (`PipelineRenderResponse`) returns `rendering_variants` (a list of int indices) and `total_variants` — NOT a `variants` field. So `data.variants` is `undefined`, `variantStatuses` becomes `[]`, and the user sees the EmptyState component flash before polling fills in real data.

Fix approach — build initial variant statuses from the render request data rather than the response:

1. In `handleRender`, BEFORE calling the render API, build an initial `variantStatuses` array from the `selectedVariants` set:
   ```typescript
   const initialStatuses: VariantStatus[] = Array.from(selectedVariants).map(idx => ({
     variant_index: idx,
     status: "processing" as const,
     progress: 0,
     current_step: "Initializing render...",
   }));
   ```

2. After the render API call succeeds (`res.ok`), set the initial statuses AND move to step 4 in sequence:
   ```typescript
   setVariantStatuses(initialStatuses);
   setStep(4);
   ```

3. Remove the old `setVariantStatuses(data.variants || [])` line — the response doesn't have a `variants` field.

4. The polling mechanism (`startRenderPolling`) already updates `variantStatuses` with real data from `/pipeline/status/`. The initial statuses serve as immediate placeholder until the first poll response arrives (~2s later).

This ensures Step 4 renders variant cards immediately showing "processing / 0% / Initializing render..." instead of the EmptyState component.
