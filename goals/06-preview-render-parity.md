# Task: Preview/Render parity + global anti-repetition in segment assembly

Working directory: C:\obSID SRL\n8n\edit_factory (Windows). All paths below are relative to it. This is the Blipost desktop app (Electron + FastAPI backend + Next.js frontend).

## Background (verified by prior audit — trust these findings)

The pipeline Step 3 preview assembles video segments per SRT phrase, then merges short consecutive phrases into "merge groups" (min_segment_duration=3.0s default). Two critical defects:

**Defect 1 — Preview ≠ Render.** `match_srt_to_segments` (app/services/assembly_service.py:634) assigns a segment PER PHRASE; merge groups are computed only afterwards (`preview_matches` builds `match_group_map` around line 2819, and `build_timeline` line ~993 merges again independently). The frontend preview player (frontend/src/components/timeline-editor.tsx) plays each phrase's own segment (counter shows e.g. "13/25"), while the final render's `build_timeline` picks ONE representative per merged group and stretches it. Result: a merge-group card labeled "#6-8" plays 2-3 different clips in preview but only 1 in the final render, and when those clips overlap in source time the user sees near-identical footage repeated within a second.

**Defect 2 — Repetition.** The only hard anti-repeat rule is "not identical to the immediately previous pick" (line ~843). No usage cap, no cooldown window, and no awareness that segments are generated by a sliding window with 30-50% overlap over the same source video (app/services/video_processor.py analyze_full_video) — so two "different" segments 10-13s and 12-15s of the same source are visually the same scene but count as diverse. Also: the ultra-rapid intro in `build_timeline` (~line 1074) picks 4 micro-segments from the same pool that reappear in the body, and the end gap-fill reuses segments, both outside the recency ledger.

## Required changes

### 1. One segment per merge group, shared by preview and render (parity invariant)

Compute merge groups FIRST (from SRT phrase durations + min_segment_duration — same grouping math already in preview_matches ~2819), then assign ONE segment per GROUP (scoring can stay as-is, applied at group level). All phrases in a group carry the same segment_id in the returned matches, so:
- the frontend preview player plays exactly the clip sequence the render will produce;
- `build_timeline`'s representative choice becomes a no-op confirmation, not a divergent decision.

Keep pinned assignments (pinned_assignments) and the timeline-editor swap-propagates-to-group behavior working (frontend already propagates swaps across a merge group, timeline-editor.tsx ~line 980). Preserve the ai_smart preset path (group-level: pick the AI assignment of the group's longest phrase or majority). Do NOT change the public response shape of preview_matches (matches still per-phrase with merge_group/merge_group_duration annotations) — only the invariant "same group ⇒ same segment_id" is new.

### 2. Visual-overlap-aware global anti-repetition

- Define "visual cluster": segments from the same source_video_id whose [start_time, end_time] intervals overlap (directly or transitively) belong to one cluster. Compute clusters once per assembly from segments_data.
- Hard constraint: a cluster must not reappear within a cooldown window of the output timeline (suggest: cooldown_seconds = min(10.0, audio_duration / 3), tunable constant) — relax progressively (halve cooldown, then drop it) only when the eligible pool is empty, and record that relaxation happened.
- The intro micro-segments and end gap-fill must participate in the same ledger: intro picks count as "used at t=0..2s", gap-fill respects the cooldown too.
- Keep determinism per variant_index (shuffle preset stays seeded).

### 3. Insufficient-material warning

When assembly had to relax the cooldown or reuse clusters (pool too small vs. number of groups), add a `variety_warning` field to the preview_matches return dict, e.g. `{"level": "low_variety", "unique_clusters": N, "slots": M, "message": "..."}` — and surface it in Step 3 UI (frontend/src/app/pipeline/components/step3-preview.tsx) as a visible amber warning banner on the variant card (English copy, e.g. "Source material is too short for a varied video — N unique scenes for M slots. Add more source videos or shorten the script."). null/absent when variety is fine.
